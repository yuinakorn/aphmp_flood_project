## 0. บทสรุปผู้บริหาร (อ่านอันเดียวพอ)

HEART กับระบบเรา **หน้าตาคล้ายกัน** (สุขภาพ + ภัยพิบัติ + GIS + ศูนย์พักพิง + JHCIS + บริบทราชการไทย) แต่เป็นผลิตภัณฑ์ **คนละชนิด**:

- **HEART = สมุดบันทึกการอพยพ (evacuation logbook)** — เปิดใช้ *หลัง* น้ำมา เพื่อบันทึกว่าใครอพยพ ใครถูกส่งต่อไปไหน ศูนย์ไหนเต็ม สวยและใช้ง่าย แต่ **ระหว่างที่ไม่มีภัยพิบัติ มันไม่มีเหตุผลให้เปิด → ข้อมูลตาย** พอน้ำมารอบใหม่ต้องกรอกใหม่ตั้งแต่ศูนย์
- **ระบบเรา = ทะเบียนสุขภาพชุมชนที่มีชีวิต ใช้ทั้งปี แล้ว"กลายร่าง"เป็นศูนย์บัญชาการ** — อสม./รพ.สต. ใช้ 360 วัน/ปี ดูแลกลุ่มเปราะบาง **ข้อมูลจึงสดอยู่แล้วตอนน้ำมา** น้ำท่วมเป็น *โหมด* ไม่ใช่ *ตัวผลิตภัณฑ์*

**นี่คือความได้เปรียบเชิงโครงสร้างที่ HEART ลอกไม่ได้** — เขาลอกหน้าจอเราได้ แต่ลอก "เหตุผลที่ต้องเปิดใช้ทุกวัน" ไม่ได้

**กลยุทธ์ 2 แนวรบ:**
1. **ปิดช่องว่าง UX** — ยอมรับตรงๆ ว่า HEART "ใช้ง่ายตอนฉุกเฉิน" กว่าเราในบางจุด **ลอกสิ่งที่เขาทำดีกว่า** (7 อย่าง §3) อย่าหยิ่ง
2. **กดจุดแข็งเชิงโครงสร้าง** — ทำสิ่งที่ HEART ทำไม่ได้เพราะมันเป็นแค่ logbook (8 อย่าง §4) ให้ *เห็นชัดกลางเวที demo*

**ประโยคขายเดียว:**
> "HEART ติดตามการอพยพ — **เราป้องกันการตาย** และเรารู้จักผู้ป่วยทุกคนตั้งแต่ก่อนน้ำจะมา"
> *They track the evacuation. We prevent the deaths — and we knew every patient before the water rose.*

## 1. HEART ทำอะไรได้ดี (ของจริง ไม่เข้าข้างตัวเอง)

จาก prototype 6 หน้า — งานออกแบบข้อมูล (information design) ของเขาดีจริง:

| หน้า | สิ่งที่ทำได้ดี |
| --- | --- |
| **ภาพรวม (Overview)** | 5 stat tile สีตามความหมาย + **donut (สถานะ)** + **bar (กลุ่มเปราะบาง)** + **"เคสร้อน"** + **"ศูนย์ใกล้เต็ม"** → ผู้บัญชาการเข้าใจสถานการณ์ใน 3 วินาที |
| **แผนที่ GIS** | filter panel เป็น chip สีจัดกลุ่มชัด, หมุด cluster, โซนเสี่ยงเป็นพื้นที่แรเงา, legend ล่างซ้าย |
| **รายบุคคล & ส่งต่อ** | **master-detail แน่นมาก**: ค้นหา → เลือกคน → เห็น identity + tag + **ไทม์ไลน์** + **ฟอร์มส่งต่ออยู่ข้างกัน** → จบ loop ในจอเดียว |
| **ศูนย์พักพิง** | **การ์ด occupancy + แถบความจุ** (2/280, 5/180...) อ่าน "ศูนย์ไหนใกล้เต็ม" ทันที + จัดกลุ่มคนตามประเภท + ปุ่ม action inline (ดู/ส่งต่อ/กลับบ้าน) |
| **เพิ่ม/แก้ไขข้อมูล** | **ค้น JHCIS ด้วยเลข 13 หลัก** ก่อนกรอก, ฟอร์ม 4 คอลัมน์, checkbox กลุ่มเปราะบางสีกำกับ, consent PDPA |
| **ตั้งค่า** | tab ครบ (ศูนย์/รพ./เหตุการณ์/เจ้าหน้าที่/พื้นที่/**การเชื่อมต่อ API**) — การ์ด API (JHCIS, HIS, ปภ., สปสช., LINE OA, SMS) **เล่าเรื่อง "เสียบเข้า ecosystem ชาติ"** ได้สวย |

**แพตเทิร์น UX เด่นที่สุด 2 อย่าง:**
1. **Action banner แดงค้างที่ระบุชื่อ-ตัวเลข**: "ผู้ป่วยติดเตียงในพื้นที่เสี่ยงสูงที่ยังไม่อพยพ **5 ราย** → ดูรายชื่อ" — เปลี่ยน "โหมดวิกฤต" ให้กลายเป็น **"งานถัดไปที่ต้องทำเดี๋ยวนี้"**
2. **เมนูซ้ายเห็นตลอด 6 ปุ่ม** — กดปุ่มเดียวถึงทุกหน้า ไม่ต้องคิด

## 3. แนวรบ 1 — ลอกให้หมดเขี้ยว (7 อย่างที่ HEART ทำดีกว่า)

> หลัก: ลอก *แพตเทิร์น* ไม่ใช่ *หน้าตา* (เราสวยพอแล้ว) — เอามาวางบนโมเดลข้อมูลที่ลึกกว่าของเรา

### S1 · เมนูซ้ายเห็นตลอด แทน dropdown เดียว 🔥 (ถูก + ได้ผลสูงสุด)
**ปัญหาจริง:** ทุกเมนู admin ซ่อนใน dropdown "ผู้ดูแลระบบ" ([Masthead.tsx:155](src/components/shell/Masthead.tsx#L155)) + โผล่เฉพาะ `lg:` → ตอนตกใจต้อง คลิก→กวาดตา→หา→คลิก
**แก้:** rail ซ้ายถาวร (หรือ top-nav เห็นตลอด) 5–6 ปลายทางหลัก แต่ละอันคลิกเดียวถึง พร้อม icon+label+count chip
**เมนูหลักที่เสนอ (เลียน HEART แต่เป็นของเรา):** ภาพรวม · แผนที่ · กลุ่มเปราะบาง(บ้าน) · ศูนย์พักพิง · ทีม/คำร้อง · ตั้งค่า
**ไฟล์:** `src/components/shell/Masthead.tsx` → แตกเป็น `AppSidebar.tsx` (หรือ persistent nav)

### S2 · Action banner ที่ "ระบุชื่อ-ตัวเลข-ปุ่ม" 🔥
**ปัญหา:** `IncidentBanner` ของเราบอก *โหมด* (แดง/เหลือง/เทา) แต่ไม่บอก *งานถัดไป*
**แก้ (และเหนือกว่า — ดู L1):** banner = "ผู้ป่วยพึ่งพาออกซิเจนในเขตน้ำท่วมที่ยังไม่อพยพ **N ราย** → เปิดคิว" โดย N มาจากการ *คำนวณ* ไม่ใช่ตัวเลขนิ่ง
**ไฟล์:** `src/components/shell/IncidentBanner.tsx`

### S3 · มุมมอง "ภาพรวมผู้บัญชาการ" (comprehension layer)
**ปัญหา:** DESIGN.md เราตั้งใจ *ปฏิเสธ* stat-card grid (เลือก status ribbon austere แบบ Linear) — ดีสำหรับ operator ไล่คิว แต่ **commander + เจ้าหน้าที่มือใหม่ + คนดู demo** อ่าน ribbon ช้ากว่าเห็น donut/bar
**แก้ (ประนีประนอมกับ DESIGN อย่างมีวินัย):** ทำ **landing ตามบทบาท**
- **operator (อสม./รพ.สต. ไล่งาน)** → worklist เดิม (พระเอกเหมือนเดิม)
- **commander (EOC/admin)** → หน้า "ภาพรวม" ใหม่: ribbon + donut สถานะ + bar กลุ่มเปราะบาง + "เคสร้อน" (= คิว L1) + "ศูนย์ใกล้เต็ม"
→ ไม่ขัด DESIGN เพราะ worklist ยังเป็น *action surface*; เพิ่ม comprehension *เฉพาะ role ที่ต้องการภาพรวม*
**ไฟล์:** หน้าใหม่ `/admin/overview` หรือ segment ใหม่ใน EOC; chart เบาๆ (donut/bar ทำด้วย SVG/CSS ไม่ต้องลง lib หนัก)

### S4 · Master-detail รายคน + timeline + ฟอร์ม ในจอเดียว
**ของ HEART:** ซ้าย=ลิสต์มี severity stripe, ขวา=detail+timeline+ฟอร์มส่งต่อ
**แก้ (และเหนือกว่า — ดู L2):** ของเราเป็น **household master-detail**: เลือกบ้าน → เห็นสมาชิกทุกคน + timeline ราย*คน* + ฟอร์มเยี่ยม/ส่งต่อข้างกัน
**ไฟล์:** ยก `FieldActionSheet` + `/admin/vulnerable` + `/admin/family-folder` มารวมเป็น layout master-detail

### S5 · การ์ดศูนย์พักพิงลีดด้วยแถบความจุ
**ของ HEART:** แถวการ์ด occupancy + แถบ % เต็ม อ่านทันที
**แก้ (และเหนือกว่า):** เพิ่ม **"ความพร้อมด้านสุขภาพ"** บนการ์ด — รับติดเตียงได้กี่เตียง? มีออกซิเจนไหม? (เรามี zone/health-readiness ใน schema แล้ว) → เขาอ่านหัวคน เราอ่านหัวคน *+ เตียงพยาบาล*
**ไฟล์:** `/admin/shelters/page.tsx` ให้ลีดด้วยแถว card occupancy

### S6 · หน้า "การเชื่อมต่อ" เล่าเรื่อง ecosystem
**ของ HEART:** การ์ด JHCIS/HIS/ปภ./สปสช./LINE/SMS (mock แต่เล่าเก่ง)
**แก้ (และเหนือกว่า):** ของเราทำ **ของจริง 2 เส้นที่สำคัญ** — JHCIS (กันกรอกซ้ำ) + HIS discharge (รู้ว่าใครฟอกไต/ใช้ O2) + outbound LINE/SMS แจ้งผู้ดูแล → **ของจริงชนะ mock**
**ไฟล์:** หน้า `/admin/integrations` + เริ่มที่ JHCIS lookup ใน intake

### S7 · ส่งต่อเป็น verb ชั้นหนึ่ง (the "R" ใน HEART)
**ของ HEART:** บันทึกส่งต่อ (ปลายทาง+ประเภท+ผู้บันทึก) โชว์บน timeline
**แก้ (และเหนือกว่า — ดู L3):** ของเรา **ปิด loop** — ส่งต่อแล้วต้องมี *ยืนยันถึงปลายทาง* ไม่งั้น escalate

## 4. แนวรบ 2 — เหยียบจุดที่ HEART ลอกไม่ได้ (8 หมัดน็อก)

> แต่ละอันคือสิ่งที่ HEART ทำไม่ได้ "เว้นแต่จะกลายร่างเป็นเรา" (ทะเบียนครัวเรือนใช้ทั้งปี + backend จริง)

### L1 · คิว "ใครจะตายก่อน" (Survivability Triage) 🏆
HEART โชว์ตัวเลขนิ่ง "5 ราย" — เรา **คำนวณคิวเรียงลำดับ** จาก:
`คะแนน = พึ่งพาอุปกรณ์พยุงชีพ (O2/ฟอกไต/กันชัก = ชั่วโมงถึงอันตราย) × ระยะใกล้น้ำ (จาก telemetry สด) × เวลาตั้งแต่ติดต่อล่าสุด × ไม่มีผู้ดูแล × ความพร้อมศูนย์ปลายทาง`
banner → คิวที่ *เรียงตามใครตายก่อน* + **ทีมกู้ภัยที่ใกล้สุดถูกเสนอให้แล้ว**
**ทำไม HEART ลอกไม่ได้:** ไม่มีโมเดลอุปกรณ์พยุงชีพ, ไม่มีน้ำสด, ไม่มีทีม, ไม่มีประวัติติดต่อทั้งปี

### L2 · อพยพเป็น "ครัวเรือน" ไม่ใช่ "รายคน" 🏆
"หมุด = บ้าน" — detail = **การ์ดครัวเรือน**: สมาชิกทุกคน + ใครต้องใช้อะไร + ผู้ดูแล + เบอร์ทุกคน + **"รายการต้องหยิบ"** (ถัง O2, น้ำยาฟอกไต, ยากันชัก 3 วัน)
หนึ่งเที่ยวเรือ = หนึ่งการ์ด **HEART เป็น row รายคน → จะหยิบ 4 คนในบ้านเดียวซ้ำ และอาจทิ้งผู้ดูแลไว้**

### L3 · ส่งต่อแบบปิด loop — "ส่งแล้วไม่ถึง" 🏆
HEART บันทึกส่งต่อแล้วจบ — เราเก็บ state: ขอ → กำลังนำส่ง → **ยืนยันถึงโดยปลายทาง** (ไม่งั้น escalate)
**จอเด็ด:** รายชื่อ "ส่งต่อแล้วแต่ไม่มีใครยืนยันว่าถึง" = คนที่หล่นจากรอยร้าว — logbook ไม่มีวันจับได้

### L4 · น้ำสดขับความเสี่ยงอัตโนมัติ 🏆
P.67→P.1 ข้ามเกณฑ์ → ระบบ recompute บ้านที่เพิ่งเข้ารัศมีเสี่ยง → ดัน "ผู้ป่วยติดเตียง 3 รายเพิ่งเข้าเขตเสี่ยง" เข้าคิว
HEART โซนเป็นสี่เหลี่ยมวาดมือ — น้ำขึ้นแล้วโซนไม่ขยับ

### L5 · แบ่งโซนทีมกู้ภัย — กันช่วยซ้ำ/บ้านถูกทิ้ง 🏆
บทเรียนสนามจริง: ทีมกระจุกพื้นที่ง่าย ลืมพื้นที่ไกล — เราจัดโซน + โชว์ช่องโหว่ coverage **HEART ไม่มีโมเดลทีม**

### L6 · ใช้ทั้งปี → ความพร้อม (คูเมืองหน้าแล้ง) 🏆🏆
โหมดปกติ = worklist ป้องกัน: หมู่บ้านเรียงตาม "ไม่ได้เยี่ยมนานสุด × จำนวนเปราะบาง × ประวัติน้ำท่วม"
**พอน้ำมา ทะเบียนเราสด — HEART เริ่มจากศูนย์ทุกครั้ง** นี่คือคูเมือง: HEART ว่างเปล่าจนภัยมา เราเดินเข้าหาภัยทั้งที่รู้จักผู้ป่วยติดเตียงทุกคนแล้ว

### L7 · Sit Rep ราชการคลิกเดียว 🏆
มี `getSitRepAuto()` แล้ว — ปภ./สสจ. ต้องการรายงานสถานการณ์มาตรฐาน เราออกให้อัตโนมัติจาก ops สด (+ override มือ)
**ล็อกเชิงราชการ:** คนอนุมัติงบต้องการรายงาน เรายื่นให้ — HEART ไม่มี output ราชการ

### L8 · Field PWA ออฟไลน์สำหรับ อสม. 🏆
คนเก็บข้อมูลจริงลุยน้ำถือมือถือไม่มีสัญญาณ — HEART เป็น dashboard บนโต๊ะออฟฟิศ
ถ้า field capture ของเราออฟไลน์ได้+sync ได้ **เราเป็นเจ้าของชั้นข้อมูลที่จุดเก็บ** — ซึ่งคือเหตุผลที่ข้อมูลเราสดและเขาตาย

## 7. ข้อเสนอแก้ DESIGN.md / PRODUCT.md (delta — ยังไม่ลงมือจนกว่าจะอนุมัติ)

> เอกสารทั้งสองเพิ่งอัปเดต 31 พ.ค. และแข็งแรง — เสนอ *เพิ่ม* ไม่ใช่เขียนทับ

**DESIGN.md เพิ่ม:**
1. **Pattern: Persistent app nav** — rail ซ้าย/top-nav เห็นตลอด 5–6 ปลายทาง + count chip (เลิกซ่อนใน dropdown); mobile = bottom-nav
2. **Pattern: Actionable crisis banner** — banner ต้องระบุ *จำนวน+ปุ่มไปคิว* ไม่ใช่แค่สีโหมด
3. **Pattern: Role-based landing** — operator→worklist, commander→comprehension dashboard (donut/bar/เคสร้อน/ศูนย์ใกล้เต็ม); ระบุชัดว่า "comprehension layer ใช้เฉพาะ commander/overview ไม่ใช่ทุกหน้า ops" → คง worklist เป็นพระเอกของหน้า ops
4. **Pattern: Household master-detail** — detail = การ์ดครัวเรือน (สมาชิก+อุปกรณ์+bring-list) ไม่ใช่ row รายคน
5. **Component เพิ่ม:** `.gx-capacity-bar` (แถบความจุศูนย์), `.gx-donut` / `.gx-bar-track` (chart เบา SVG/CSS), `.gx-timeline`

**PRODUCT.md เพิ่ม:**
1. **คู่แข่งอ้างอิง:** HEART = evacuation logbook (reactive) ↔ เรา = living registry (proactive) + ประโยคขาย §0
2. **เสาหลักใหม่/ชัดขึ้น:** Survivability triage (L1), Closed-loop referral (L3) — ระบุเป็น pillar
3. **Anti-reference เพิ่ม:** "อย่าเป็น logbook ที่ข้อมูลตายระหว่างภัย"
